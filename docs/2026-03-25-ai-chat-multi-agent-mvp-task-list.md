# MVP 开发任务清单：AI Chat 多 Agent 分析工作台

- **文档版本**：V1.0
- **日期**：2026-03-25
- **适用范围**：MVP

---

# 1. 文档目标

本文档用于将 AI Chat 多 Agent 分析工作台的 MVP 范围拆解为可执行的研发任务，便于产品、设计、研发和测试协同推进。

---

# 2. 里程碑拆分

## M1：跑通最小闭环
目标：
- 可创建 Session
- 可上传文件
- 可聊天
- 可生成报告
- 可保存数据

## M2：打磨双 Agent 体验
目标：
- 用户能明显感知两个 Agent
- 页面结构更清晰
- 上传、报表、对话三者关系更清楚

## M3：增强稳定性与可演示性
目标：
- 错误处理完整
- 空状态完善
- Demo 可顺畅演示

---

# 3. 任务分组

# A. 产品与信息架构任务

## A1. 确认 MVP 功能边界
- **优先级**：P0
- **内容**：
  - 锁定首版只做两个 Agent
  - 锁定手工上传为唯一数据入口
  - 锁定报表为手动触发
  - 锁定首版为通用分析报告
- **产出**：
  - PRD 定稿
  - 范围说明定稿
- **验收标准**：
  - 所有人对 MVP 范围无歧义

## A2. 定义 Session 使用规则
- **优先级**：P1
- **内容**：
  - 明确一个 Session 是否对应一个分析主题
  - 明确是否允许混放多个无关数据集
- **验收标准**：
  - 页面文案和产品说明一致

---

# B. 前端页面任务

## B1. Workspace 页面结构梳理
- **优先级**：P0
- **关键文件**：`components/workspace/workspace-shell.tsx`
- **内容**：
  - 保持三栏结构
  - 明确左中右三部分职责
  - 统一区域标题与信息层级
- **验收标准**：
  - 页面结构清晰
  - 用户能快速找到聊天、上传、报表入口

## B2. 强化双 Agent 展示
- **优先级**：P0
- **关键文件**：
  - `components/workspace/workspace-shell.tsx`
  - `components/workspace/message-list.tsx`
- **内容**：
  - 明确展示“通用对话 Agent”
  - 明确展示“报表 Agent”
  - 对消息来源做身份标识
- **验收标准**：
  - 用户能一眼分辨当前内容由哪个 Agent 生成

## B3. 优化聊天区交互
- **优先级**：P0
- **关键文件**：
  - `components/workspace/message-composer.tsx`
  - `components/workspace/message-list.tsx`
- **内容**：
  - 输入状态优化
  - 发送中状态
  - 错误提示
  - 空状态提示
- **验收标准**：
  - 空输入不可提交
  - 失败时提示明确
  - 消息流反馈顺畅

## B4. 优化上传区体验
- **优先级**：P1
- **关键文件**：`components/workspace/upload-list.tsx`
- **内容**：
  - 明确支持格式
  - 展示摘要 / 字段预览
  - 展示上传失败原因
  - 达到上限时给出提示
- **验收标准**：
  - 用户知道上传了什么
  - 用户知道为什么失败

## B5. 优化报表区体验
- **优先级**：P0
- **关键文件**：`components/workspace/report-panel.tsx`
- **内容**：
  - 明确“生成报告”入口
  - 无上传时禁用并提示
  - 报表列表更像成果区
  - 展示标题、摘要、生成时间
- **验收标准**：
  - 用户知道如何生成报告
  - 报告与普通消息有明显区别

---

# C. 会话与状态管理任务

## C1. Session 列表与当前状态同步
- **优先级**：P0
- **关键文件**：
  - `components/workspace/workspace-shell.tsx`
  - `lib/storage/repository.ts`
- **内容**：
  - Session 更新时间同步
  - 最新活跃会话排序
  - 最近活跃 Agent 展示
- **验收标准**：
  - 发送消息、上传、生成报告后 Session 状态正确刷新

## C2. Session 标题生成规则确认
- **优先级**：P1
- **关键文件**：`lib/storage/repository.ts`
- **内容**：
  - 是否继续沿用“首条消息自动命名”
  - 是否需要更清晰标题截断规则
- **验收标准**：
  - Session 标题表现稳定可读

---

# D. Chat 能力任务

## D1. 通用对话 Agent 主链路稳定化
- **优先级**：P0
- **关键文件**：`app/api/chat/route.ts`
- **内容**：
  - 校验请求参数
  - 获取 Session 上下文
  - 写入用户消息
  - 生成回复
  - 写入 assistant 消息
- **验收标准**：
  - 正常对话链路稳定
  - 消息写入顺序正确

## D2. 强化对上传内容与报告的引用
- **优先级**：P1
- **关键文件**：
  - `lib/agents/anthropic.ts`
  - `app/api/chat/route.ts`
- **内容**：
  - 让对话 Agent 更稳定利用：
    - 上传摘要
    - 历史报告
  - 增强追问承接能力
- **验收标准**：
  - 用户上传后提问，回答明显更贴近上传内容
  - 有历史报告时可引用报告结论

## D3. 增加“建议生成报告”提示
- **优先级**：P2
- **关键文件**：
  - `lib/agents/anthropic.ts`
  - 前端文案层
- **内容**：
  - 当问题明显偏总结/汇报时，Agent 给出建议
- **验收标准**：
  - 不自动触发报表
  - 只做自然引导

---

# E. Report 能力任务

## E1. 报表生成主链路稳定化
- **优先级**：P0
- **关键文件**：`app/api/reports/route.ts`
- **内容**：
  - 校验 sessionId
  - 校验是否存在上传
  - 读取上传和历史报告
  - 生成报告
  - 写入 report artifact
  - 写入 report message
- **验收标准**：
  - 报表链路稳定
  - 无上传时明确拒绝生成

## E2. 固化通用分析报告结构
- **优先级**：P0
- **关键文件**：`lib/agents/anthropic.ts`
- **内容**：
  - 统一首版报告结构：
    - 概览
    - 关键发现
    - 风险 / 异常
    - 原因分析
    - 建议动作
- **验收标准**：
  - 每次生成报告结构基本一致
  - 适合演示与复用

## E3. 报告与源文件关系增强
- **优先级**：P1
- **关键文件**：
  - `app/api/reports/route.ts`
  - `components/workspace/report-panel.tsx`
- **内容**：
  - 明确展示报告基于哪些上传内容生成
- **验收标准**：
  - 用户能知道报告来源

---

# F. Upload 能力任务

## F1. 上传校验逻辑确认
- **优先级**：P0
- **关键文件**：
  - `lib/uploads/parse-upload.ts`
  - `app/api/uploads/route.ts`
- **内容**：
  - 文件类型校验
  - 文件大小校验
  - 上传数量校验
- **验收标准**：
  - 超限和非法格式都有明确错误

## F2. 上传摘要展示优化
- **优先级**：P1
- **关键文件**：
  - `lib/uploads/parse-upload.ts`
  - `components/workspace/upload-list.tsx`
- **内容**：
  - 提高 `summary`、`schemaPreview` 可读性
- **验收标准**：
  - 用户能快速理解上传内容是什么

## F3. 空文件与脏数据处理
- **优先级**：P1
- **内容**：
  - 空文件拒绝
  - 极少字段、空表头、异常文本时的提示优化
- **验收标准**：
  - 上传异常反馈更可理解

---

# G. 持久化与数据模型任务

## G1. 核心模型稳定性确认
- **优先级**：P0
- **关键文件**：`lib/domain/types.ts`
- **内容**：
  - 明确首版只保留：
    - `conversation`
    - `report`
  - 保持 Session / Message / Upload / Report 主模型稳定
- **验收标准**：
  - 模型不反复变更
  - 前后端使用一致

## G2. Repository 行为回归检查
- **优先级**：P0
- **关键文件**：`lib/storage/repository.ts`
- **内容**：
  - 检查：
    - message 写入
    - upload 写入
    - report 写入
    - session 更新
    - 排序逻辑
- **验收标准**：
  - 刷新页面后数据一致
  - 最新会话排序正确

---

# H. 错误处理与空状态任务

## H1. 全链路错误提示统一
- **优先级**：P0
- **内容**：
  - 聊天失败
  - 上传失败
  - 报表失败
  - 无上传时报表不可生成
- **验收标准**：
  - 错误提示位置统一
  - 文案直接可懂

## H2. 空状态设计补齐
- **优先级**：P1
- **内容**：
  - 空 Session
  - 无上传
  - 无报告
  - 首次进入页面
- **验收标准**：
  - 用户知道下一步该做什么

---

# I. 测试与验证任务

## I1. 端到端主链路验证
- **优先级**：P0
- **内容**：
  1. 创建 Session
  2. 上传 CSV
  3. 提问
  4. 获得回复
  5. 生成报告
  6. 继续追问
  7. 刷新恢复
- **验收标准**：
  - 主链路完整跑通

## I2. 边界场景验证
- **优先级**：P0
- **内容**：
  - 空文件
  - 不支持格式
  - 超文件大小
  - 超上传数量
  - 无上传生成报告
  - 模型服务不可用
- **验收标准**：
  - 都有合理反馈

## I3. Demo 场景准备
- **优先级**：P1
- **内容**：
  - 准备 2~3 组演示数据
  - 准备标准演示路径
- **验收标准**：
  - Demo 可稳定复现

# 6. 最小 Sprint 切法建议

## Sprint 1
- 跑通最小闭环
- 保证上传、聊天、报表都能稳定工作

## Sprint 2
- 强化双 Agent 展示
- 优化上传和报表体验
- 固化通用分析报告结构

## Sprint 3
- 完善错误处理
- 完善空状态
- 准备 Demo 数据与演示流程

---

# 7. 结论

MVP 开发建议遵循以下原则：
- 先完成最小闭环
- 再优化双 Agent 感知
- 再强化稳定性与演示能力
- 始终围绕“上传—提问—生成报告—继续分析”这一主路径推进
