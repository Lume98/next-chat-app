# 1. 产品概述

## 1.1 项目背景
当前用户在处理业务数据时，通常需要在多个工具之间切换：先整理 CSV/Excel，再用聊天工具提问，再手动整理成报告。流程割裂，效率低，且分析结果不易沉淀和复用。

本项目拟建设一个面向单人使用场景的 AI 分析工作台，先通过 **多 Agent 但轻编排** 的方式，聚焦两类最核心能力：

1. **通用对话 Agent**：负责问答、解释、探索式分析
2. **报表 Agent**：负责基于上传数据生成结构化分析报告

产品首期不追求复杂自治能力，不接企业系统，不做权限与协作，优先验证“上传—分析—报表”这一最小闭环是否成立。

## 1.2 产品定位
这是一个 **面向单用户的数据分析 AI 工作台**，核心价值是：

- 支持用户手工上传业务文件
- 支持围绕文件内容进行自然语言问答
- 支持一键生成可沉淀的分析报告
- 支持在同一工作区内持续追问和复用结果

## 1.3 产品目标
在 MVP 阶段，实现一个可演示、可试用、可扩展的双 Agent 产品基础，让用户能够完成：

**上传文件 → 与 AI 对话分析 → 生成结构化报告 → 基于报告继续追问**

# 2. 产品目标与衡量

## 2.1 业务目标
- 验证用户是否认可“双 Agent 分析工作台”的产品形态
- 验证手工上传数据驱动的 AI 分析闭环是否具备实际价值
- 为后续增加更多 Agent、模板、数据接入方式打下基础

## 2.2 用户目标
用户希望能够：
- 快速上传 CSV/Excel/文本文件
- 用自然语言直接提问
- 获得可理解的分析回答
- 一键生成结构化报告
- 不需要复杂配置即可开始使用

## 2.3 MVP 成功标准
满足以下条件视为 MVP 成立：

1. 用户可以创建并进入一个 Session
2. 用户可以上传 CSV / Excel / 文本文件
3. 用户可以通过通用对话 Agent 发起分析问题
4. 系统可返回基于上传内容的回答
5. 用户可以手动触发报表 Agent
6. 系统可以生成一份结构化通用分析报告
7. 用户可以基于报告继续追问
8. 上述内容都能保存在当前 Session 中

# 3. 目标用户

## 3.1 核心用户
- 个人分析师
- 运营人员
- 销售管理人员
- 财务或经营分析人员
- 需要快速“上传数据 + 问答 + 出报告”的单兵用户

## 3.2 用户特征
- 多数以单人使用为主
- 手中已有 CSV/Excel/文本资料
- 对传统 BI/报表工具使用门槛敏感
- 更希望通过自然语言完成分析与总结

# 4. 产品范围

## 4.1 In Scope（本期范围）

### A. Session 工作区
- 用户可以创建和进入 Session
- 一个 Session 对应一个分析主题或任务
- Session 内聚合：
  - 消息记录
  - 上传文件
  - 报表结果

### B. 通用对话 Agent
- 默认主交互入口
- 支持自然语言问答
- 可结合以下上下文回答：
  - 当前消息历史
  - 上传文件摘要
  - 已生成报告内容

### C. 报表 Agent
- 由用户手动触发
- 基于当前 Session 已上传文件生成一份通用分析报告
- 报表作为独立成果保存在 Session 下

### D. 文件上传
支持手工上传：
- CSV
- Excel
- Text / Markdown

系统对上传内容做轻量解析，提取：
- 文件摘要
- 字段/结构预览
- 可供分析使用的文本上下文

### E. 报表展示与复用
- 用户可查看当前 Session 的历史报告
- 报告与源文件存在关联关系
- 后续对话可以引用报告内容

## 4.2 Out of Scope（本期不做）
- 多人协作
- 权限系统 / RBAC
- 企业系统对接
- 自动拉取外部数据
- 向量检索 / RAG
- 复杂 Agent 编排
- 自动后台任务链
- 流式响应
- 数据库 / 对象存储升级
- 审计、审批、计费等企业级能力

# 5. 产品形态设计

## 5.1 Agent 形态
本期对外明确呈现为 **双 Agent**：

- **通用对话 Agent**
- **报表 Agent**

其中：
- 默认由通用对话 Agent 承接用户问题
- 报表 Agent 通过明确按钮或动作触发
- 系统不做自动切换和隐式执行

## 5.2 报表触发方式
采用 **手动触发**：
- 用户点击“生成报告”
- 或显式发起生成动作

原因：
- MVP 更简单可控
- 降低错误触发概率
- 降低多 Agent 协调复杂度

## 5.3 报表形式
首版统一输出 **通用分析报告**，不引入多模板机制。

建议报告结构：
1. 概览
2. 关键发现
3. 异常 / 风险
4. 原因分析
5. 建议动作

# 6. 核心用户流程

## 6.1 主流程
1. 用户创建或进入一个 Session
2. 用户上传 CSV / Excel / 文本文件
3. 用户与通用对话 Agent 进行问答
4. 系统基于上传内容与上下文返回分析结果
5. 用户点击“生成报告”
6. 报表 Agent 输出结构化分析报告
7. 用户继续基于报告进行追问和扩展分析

## 6.2 交互原则
- 默认入口始终是通用对话 Agent
- 报表 Agent 负责“沉淀结果”，不是主对话入口
- 两个 Agent 分工清晰，避免职责重叠
- 用户始终能感知当前输出来自哪个 Agent

# 7. 功能需求

## 7.1 Session 管理
### 功能说明
系统支持用户按分析主题组织工作内容。

### 需求点
- 创建 Session
- 查看 Session 列表
- 打开指定 Session
- 按更新时间排序
- 保存最近活跃 Agent 状态
- 保存最近生成报告标识

## 7.2 文件上传
### 功能说明
用户通过手工上传文件，将分析资料带入工作区。

### 需求点
- 支持 CSV、Excel、Text/Markdown
- 文件大小限制
- 文件数量限制
- 上传成功后生成摘要和结构预览
- 上传失败时返回明确错误提示

### 预期输出
每个上传文件至少具备：
- 文件名
- 类型
- 大小
- 摘要
- 字段/结构预览
- 分析文本上下文

## 7.3 通用对话 Agent
### 功能说明
负责承接绝大多数问题输入，支持探索式分析和追问。

### 需求点
- 接收自然语言输入
- 结合当前 Session 上下文生成回复
- 可参考上传内容摘要
- 可参考历史报告内容
- 输出需具备可理解性和分析导向

### 典型问题
- 这个表里有什么异常？
- 本月趋势如何？
- 哪几个字段最关键？
- 请解释这份报告里的结论

## 7.4 报表 Agent
### 功能说明
负责将当前 Session 中的上传内容沉淀为结构化输出。

### 需求点
- 用户手动触发生成
- 若无上传文件，禁止生成并提示原因
- 生成通用分析报告
- 报告应作为独立 Artifact 保存
- 报告需可在消息流中被引用

### 报告要求
- 标题清晰
- 摘要明确
- 结构固定
- 内容可读、可复用、可继续追问

## 7.5 报告展示
### 功能说明
用户可在 Session 内查看已生成的报告，并继续围绕报告互动。

### 需求点
- 展示报告列表
- 查看报告详情
- 记录报告生成时间
- 关联来源上传文件
- 支持后续对话引用报告

# 8. 信息架构与页面结构

## 8.1 页面布局建议
### 左侧
- Session 列表
- 当前工作区信息

### 中间
- 消息对话区
- 输入框
- 当前主交互区域

### 右侧
- 上传文件区
- 报表区
- 生成报表入口

## 8.2 设计原则
- 双 Agent 身份清晰
- 用户操作路径短
- 报表入口可见
- 上传、对话、报表彼此关联
- 不为 MVP 引入复杂导航层级

# 9. 非功能需求

## 9.1 性能
- 常规小文件上传后应可快速完成解析
- 对话和报表生成在合理时间内返回
- 不要求首版支持高并发

## 9.2 可用性
- 上传失败有明确提示
- 空内容、无文件、格式不支持等情况可识别
- 用户能分辨当前是聊天结果还是报表结果

## 9.3 数据存储
- MVP 阶段允许本地持久化
- 重点验证产品流程，不追求生产级存储方案

## 9.4 可扩展性
后续可扩展方向包括：
- 更多 Agent
- 更多报表模板
- 企业数据源接入
- 权限和协作
- 数据库存储
- 检索与知识增强能力

---

# 10. 风险与约束

## 10.1 风险
- 上传内容质量差会影响分析质量
- 首版无 RAG，长上下文能力有限
- 通用分析报告可能难覆盖所有行业场景
- 单用户设计不适合直接扩展为企业协作场景

## 10.2 当前约束
- 以手工上传为唯一数据入口
- 不做自动调度和复杂路由
- 仅保留两个 Agent
- 首版报告为通用结构

# 11. 验证方案

## 11.1 主链路验证
1. 创建 Session
2. 上传 CSV 文件
3. 发起普通问题
4. 获得通用对话 Agent 回答
5. 手动触发生成报告
6. 获得报表 Agent 输出
7. 基于报告继续提问
8. 关闭再打开后，内容仍可恢复

## 11.2 异常验证
- 空文件上传
- 超出大小限制
- 不支持格式
- 无上传文件直接生成报表
- 模型服务不可用时的降级提示

## 11.3 体验验证
- 用户是否理解双 Agent 区别
- 用户是否能快速找到“生成报告”
- 用户是否愿意继续围绕报告追问
- 用户是否能理解上传内容与回答之间的关系

# 12. 版本规划建议

## Phase 1：双 Agent MVP 成型
- Session + 上传 + 对话 + 报表链路打通
- 明确双 Agent 角色

## Phase 2：体验增强
- 对话 Agent 提示用户何时适合生成报告
- 优化文件摘要与字段展示

## Phase 3：能力增强
- 增加更多报表结构
- 增强上下文引用能力
- 提升分析稳定性

---

# 13. 最终结论

本期 MVP 推荐定义为：

> 一个面向单人分析场景的双 Agent AI 工作台，用户通过手工上传 CSV/Excel/文本，在同一 Session 中完成自然语言分析与结构化报告生成。

其核心策略是：
- **先做最小可用双 Agent**
- **先做手工上传闭环**
- **先做单用户工作台**
- **先做通用分析报告**
- **先不做复杂协作和企业集成**
