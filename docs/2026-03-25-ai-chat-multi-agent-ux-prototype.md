# 页面与交互原型说明：AI Chat 多 Agent 分析工作台（MVP）

- **文档版本**：V1.0
- **日期**：2026-03-25
- **适用范围**：MVP

---

# 1. 文档目标

本文档用于定义 AI Chat 多 Agent 分析工作台的页面结构、信息布局、核心交互和关键状态，供产品、设计、前端开发统一理解。

---

# 2. 设计目标

MVP 页面设计目标：
- 让用户快速理解这是一个“双 Agent 工作台”
- 让用户快速完成上传、提问、生成报告三步核心路径
- 让聊天、上传、报告三者自然关联
- 保持页面轻量，不引入复杂导航和多层操作

---

# 3. 页面结构总览

MVP 采用三栏布局：

## 左栏：Session 区
用于承载工作区和历史会话信息。

内容包括：
- Session 列表
- 当前 Session 标识
- 当前 Session 最近活跃状态
- 上传/报表数量概览（可选）

## 中栏：聊天主区域
用于承载通用对话 Agent 的主交互。

内容包括：
- 当前 Agent 标识
- 消息列表
- 输入框
- 发送按钮
- 错误提示

## 右栏：上传与报表区
用于承载输入资料和结构化成果。

内容包括：
- 上传文件区域
- 上传文件列表
- 报表列表
- 生成报告按钮
- 当前报告摘要/详情

---

# 4. 页面模块定义

# 4.1 Session Sidebar
## 目标
帮助用户理解“当前在什么工作区中”。

## 模块内容
- Session 标题
- Session 列表
- 最近更新时间
- 最近活跃 Agent（可选展示）
- 报表数量/上传数量（可选）

## 交互
- 点击切换 Session
- 高亮当前 Session
- 新建 Session（如当前已有入口则沿用）

## 设计要求
- 列表清晰，不堆叠复杂信息
- 当前 Session 明确可见
- 便于用户理解一个 Session 对应一个分析任务

---

# 4.2 Agent Header / 当前角色提示
## 目标
强化双 Agent 产品认知。

## 推荐展示内容
- 当前默认主交互角色：**通用对话 Agent**
- 辅助说明：
  - 负责问答、解释、探索分析
  - 报表可在右侧手动生成

## 设计建议
- 放在聊天区顶部
- 不需要做成复杂切换器
- 首版不建议做 Tab 式强切换

---

# 4.3 Message List
## 目标
承载聊天历史和报表生成通知。

## 消息类型
- 用户消息
- 通用对话 Agent 回复
- 报表 Agent 生成结果消息

## 展示要求
- 明确区分用户/系统/Agent
- 对 `conversation` 和 `report` 的消息做身份标识
- 报表消息建议附带“查看报告”或“已生成报告”提示

## 交互要求
- 最新消息自动滚动可见
- 报错时在消息区上方展示错误提示

---

# 4.4 Message Composer
## 目标
为用户提供主输入入口。

## 组成
- 文本输入框
- 发送按钮
- 发送中状态

## 交互规则
- 空输入不可发送
- 发送后清空输入框
- 发送中按钮禁用
- 失败后显示错误提示

## 文案建议
占位文案示例：
- 输入你的问题，我会结合上传内容帮你分析

---

# 4.5 Upload Panel
## 目标
让用户快速把资料带入工作区。

## 组成
- 上传按钮 / 拖拽区域（首版可仅按钮）
- 已上传文件列表
- 上传中状态
- 上传失败提示

## 每个上传项展示
- 文件名
- 文件类型
- 文件大小
- 上传时间（可选）
- 文件摘要
- 字段/结构预览

## 交互规则
- 上传成功后立即出现在列表中
- 上传失败时明确提示原因
- 若达到上限，阻止继续上传并提示

## 文案建议
- 支持 CSV / Excel / 文本文件上传
- 上传后可直接用于问答与报告生成

---

# 4.6 Report Panel
## 目标
让用户感知“报告是结构化成果”。

## 组成
- 生成报告按钮
- 报告列表
- 当前报告摘要或详情区域
- 生成中状态

## 交互规则
- 无上传文件时，按钮禁用
- 点击生成后进入 loading
- 生成成功后：
  - 列表新增一条报告
  - 消息流插入 report agent 消息
  - 当前 Session 状态更新

## 每份报告展示信息
- 标题
- 摘要
- 生成时间
- 来源上传文件数量

## 首版建议
- 报告默认按最新优先排序
- 右侧直接展示当前选中报告摘要
- 不必首版做复杂版本对比

---

# 5. 核心用户流说明

# 5.1 用户流 1：进入工作区开始提问
1. 用户打开页面
2. 进入一个 Session
3. 在中栏看到通用对话 Agent
4. 直接输入问题
5. 获得回答

**目标**：即使未上传文件，也能先开始交流。

---

# 5.2 用户流 2：上传数据后分析
1. 用户在右侧上传文件
2. 文件出现在上传列表
3. 用户回到中栏提问
4. 通用对话 Agent 回答时结合上传上下文

**目标**：让用户感知上传内容已进入分析上下文。

---

# 5.3 用户流 3：生成报告
1. 用户在右侧点击“生成报告”
2. 按钮进入生成中状态
3. 报表生成完成
4. 右侧列表出现新报告
5. 中栏出现一条 report agent 消息

**目标**：让报表生成行为清晰、独立、可追踪。

---

# 5.4 用户流 4：基于报告继续追问
1. 用户看到报告内容
2. 用户继续在中栏提问
3. 通用对话 Agent 结合报告内容回复

**目标**：形成“报告不是终点，而是后续分析输入”的体验。

---

# 6. 页面状态设计

# 6.1 空状态
## 空 Session
- 提示用户上传文件或直接提问
- 推荐文案：
  - 先上传 CSV / Excel / 文本文件，或直接告诉我你想分析什么

## 无报告状态
- 报表区提示：
  - 上传文件后可生成结构化分析报告

---

# 6.2 Loading 状态
- 聊天发送中：输入区按钮 loading
- 上传中：上传区 loading
- 报表生成中：生成按钮 loading

要求：
- 不阻塞其他无关区域过度操作
- 状态明确可见

---

# 6.3 错误状态
需要覆盖：
- 发送消息失败
- 上传失败
- 报表生成失败
- 无上传时报表生成被拒绝

表现方式：
- 聊天区顶部错误提示
- 上传区就地提示
- 报表区就地提示

要求：
- 错误提示要说明原因
- 文案简单直接

---

# 7. 关键文案建议

## 通用对话 Agent 区
- 标题：通用对话 Agent
- 描述：负责问答、解释和探索式分析

## 报表区
- 标题：报表 Agent
- 描述：基于上传内容生成结构化分析报告

## 生成按钮
- 文案：生成报告
- loading：生成中...

## 上传区
- 标题：上传数据
- 描述：支持 CSV / Excel / 文本文件

---

# 8. 设计原则

## 8.1 清晰优先
首版最重要的是让用户理解：
- 有两个 Agent
- 一个负责对话
- 一个负责报告

## 8.2 路径最短
从进入页面到生成第一份报告，步骤尽量少。

## 8.3 成果可见
报告必须在视觉上和普通消息区别开。

## 8.4 轻量不复杂
不做复杂模式切换、不做高级配置面板、不做多级筛选。

---

# 9. 首版不建议加入的交互

以下内容建议延后：
- 自动切换 Agent
- 报表模板切换器
- 多文件筛选生成
- 拖拽式工作流
- 多标签分析空间
- 多人评论协作

---

# 10. 结论

MVP 页面与交互应围绕一个核心原则展开：

> 用最少的界面复杂度，让用户清楚地完成“上传—提问—生成报告—继续分析”的双 Agent 闭环。

首版重点不是炫技，而是：
- 角色清楚
- 操作顺畅
- 成果可见
- 路径闭环
